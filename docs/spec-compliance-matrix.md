# Symphony TypeScript implementation/compliance matrix

This document records the current local implementation posture for `symphony-ts` against the OpenAI Symphony service specification. It is intentionally conservative: local fake/demo verification, temp-home Kanban tests, and no-side-effect readiness checks are evidence for local behavior only. Hermes Kanban is the canonical work-engine direction, while direct Linear/Codex behavior is legacy compatibility and still requires reviewed authorization, bounded workflow config, private durable state, redacted receipts, and rollback evidence.

## Status legend

- **Implemented** — covered by local code and tests.
- **Implemented, gated live path** — local behavior is implemented; live use requires explicit operator authorization, bounded scope, and receipts.
- **Explicit non-goal** — not implemented by design in this package state.
- **Deferred live evidence** — local behavior is testable, but no current run in this branch proves live external API behavior.

### Legacy Codex quarantine/removal posture

The `symphony-codex-*` CLIs and the underlying `CodexAppServerRunner` surface are retained only as legacy compatibility and operator-recovery paths. A successful focused fake test or a bounded live Codex issue-run proves only that the legacy runner seam can spawn an app-server for the exact approved issue. It does **not** prove that the Kanban-first work engine (`backend.kind: hermes_kanban`) can materialize a task graph, pass a no-worker canary, or safely dispatch through the Hermes gateway. Do not cite direct Codex run receipts or focused Codex test success as evidence for Kanban readiness; cite Kanban canary, bridge, or integration receipts instead. A future reviewed removal slice may retire these CLIs and the direct runner when equivalent or better Kanban-first evidence exists for all supported workflows.

## Selected implementation-defined policies

The API-readable ledger is exported as `implementationPolicyDecisions` from `src/index.ts`. The current selected policies are:

1. Local fake/demo verification is the default trust posture.
2. Hermes Kanban is the canonical Symphony work engine. Live Kanban behavior is fail-closed without a reviewed operator packet naming exact board/profile/workspace/artifact scope and dispatch policy. Legacy live Linear/Codex behavior remains fail-closed behind digest-bound operator confirmation; exact canary scope remains preferred for legacy canaries, while any broad phase requires explicit `tracker.allow_broad_dispatch`, bounded states/per-poll/concurrency, durable state, and receipts. The `symphony-codex-*` CLIs and the underlying `CodexAppServerRunner` surface are retained only as legacy compatibility/operator-recovery paths and are not evidence of Kanban readiness; a successful direct Codex issue-run or focused fake Codex test proves only that the legacy runner seam works for the exact approved issue and does not prove that the Kanban work engine can materialize a task graph, pass a no-worker canary, or safely dispatch through the Hermes gateway. A future reviewed removal slice may retire these CLIs and the direct runner when equivalent or better Kanban-first evidence exists for all supported workflows.
3. Codex approval requests fail unless a workflow explicitly selects auto-approval.
4. Sandbox values pass through to the Codex runner using the documented Codex `0.141.0` wire-shape mapping.
5. Codex child processes scrub Linear-related service/tracker environment variables before spawn.
6. Fake/demo workspaces may use empty directories; live-readiness requires clean git-worktree source repos.
7. Invalid workflow reloads keep last-known-good runtime dependencies; valid reloads reapply default runtime dependencies.
8. `service.state_path` enables durable restart recovery and mutation idempotency keys.
9. The optional control plane is disabled by default, binds to loopback by default when enabled, exposes an unauthenticated minimal /health endpoint, and requires bearer authentication for /status and /snapshot on any non-loopback bind; non-loopback bind requires explicit opt-in and auth. Mutating endpoints always require bearer auth.
10. `backend.kind: hermes_kanban` makes Symphony a typed facade over Hermes Kanban. Symphony uses the CLI seam to create/read tasks, parse typed task links, materialize no-worker canaries, expose snapshots, and package no-worker canary readback receipts, but does not run a second dispatcher or authorize real-board/gateway dispatch. Typed task link create/unlink/readback support is fake-tested locally and is not live/shared-board apply approval.
11. `symphony-linear-kanban-bridge` is the normal Kanban-first Linear integration surface: it polls eligible Linear issues, including explicit all-visible Linear scope via `tracker.all_approved_projects` when reviewed and bounded, materializes Hermes Kanban tasks with durable idempotency keys, syncs observed Kanban completion back to Linear through configured lifecycle mutations, and never invokes the legacy Codex runner.
12. `symphony-graph-sync-diff` is a local-only GraphSync receipt artifact builder: it consumes an explicit JSON snapshot, writes a declared local `read_only_diff` receipt artifact, emits receipt-only severity/human-action metadata for operator findings, can write declared local `summary.md` and `status.json` triage artifacts, preserves suppressed-write non-actions, and does not query Linear, read or mutate Hermes Kanban, edit services/timers, or expose MCP/apply behavior.
13. `symphony-graph-sync-snapshot` is a read-only mapped live-observation capture surface: it reads bridge-ledger mappings, exact Linear issue relation state, and enriched Hermes Kanban task-link state, then writes local snapshot/receipt/status artifacts without writes or dispatch.
14. `runRecurringLinearKanbanGraphSyncTick` is a local coordinator contract for the recurring lifecycle + GraphSync loop: it runs lifecycle before GraphSync, preserves both receipts, emits `dispatch_reliance_decision` values, hard-blocks board-scope mismatch, defers on missing executable dependency edges, and does not edit services/timers or dispatch workers/gateway. `runRecurringLinearKanbanGraphSyncCanary` is the local artifact harness around that contract: it writes receipt/status/summary artifacts and suppresses dispatch-reliance probes unless the readiness decision is `allowed`. The packaged `symphony-linear-kanban-graph-sync-tick` CLI is the fake/local-readonly operator surface for this contract: it accepts `--mode fake_local_readonly`, `--workflow`, and `--artifact-root`, runs injected fake/local lifecycle and GraphSync fixtures, writes the same receipt/status/summary artifacts, exits `0/1/2/3` for PASS/error/REVIEW/BLOCK, records mutation non-action booleans as `false`, and does not construct live Linear/Hermes clients.
15. `symphony-graph-sync-status` is a read-only status/watchdog surface for the recurring GraphSync observer: it reads `last-run.json`, classifies `PASS`/`REVIEW`/`BLOCK` with a 15-minute default stale threshold, emits compact operator JSON, and does not authorize graph apply, Linear/Kanban mutation, service/timer edits, worker/gateway dispatch, push, publish, deploy, or raw secret exposure.
16. GraphSync Linear-authoritative Kanban blocking-edge materialization has both a fake-only helper and a separate gated live Kanban apply CLI. The live CLI is exact-scope and receipt-driven: it consumes a prior `read_only_diff` receipt, requires an explicit live flag, approved scope, Hermes command/home/board config, and max-created cap, creates only typed Hermes Kanban `blocks` links, verifies child-task readback, and does not imply Linear relation writes, service edits, dispatch, MCP mutation, push, publish, deploy, or PR authority.
17. GraphSync Kanban-authoritative Linear blocking-relation materialization is a separate gated live Linear apply CLI. It consumes a prior `read_only_diff` receipt, requires `LINEAR_API_KEY`, an explicit live flag, approved scope, and max-created cap, creates only missing Linear `blocks` relations with readback verification, and does not imply Hermes Kanban mutation, Linear state movement, service edits, dispatch, MCP mutation, push, publish, deploy, or PR authority.
18. Full live bidirectional sync is gated and cannot be claimed from a recurring lifecycle bridge alone. It requires a recurring DAG/GraphSync companion loop ordered with lifecycle ticks, fresh graph receipts, gated dependency-edge apply/proposal behavior in both authority directions, dependency-readiness output before dispatch reliance, and stop/rollback behavior for stale, cyclic, conflicting, incomplete, or over-cap graph state.
19. The `symphony-codex-*` CLIs and the underlying `CodexAppServerRunner` surface are retained only as legacy compatibility and operator-recovery paths. They are not the path to Kanban readiness; a successful direct Codex issue-run or focused fake Codex test does not prove that the Kanban work engine can materialize a task graph, pass a no-worker canary, or safely dispatch through the Hermes gateway. Do not cite direct Codex run receipts or focused Codex test success as evidence for Kanban readiness; cite Kanban canary, bridge, or integration receipts instead. A future reviewed removal slice may retire these CLIs and the direct runner when equivalent or better Kanban-first evidence exists for all supported workflows.

## Live validation boundary

Local fake/demo tests, temp-`HERMES_HOME` Kanban tests, no-worker canary unit tests, and no-side-effect readiness checks do **not** prove real-board Kanban worker execution or legacy live Linear/Codex behavior. Direct Codex issue-runs are compatibility evidence only and must not be claimed as Kanban work-engine evidence. Historical exact-scope canaries remain limited to their scopes. Operator-private evidence now proves one exact two-task dependent-worker path with provider-registered dependency enforcement, but it does not prove broad recurring service operation or that every long-running gateway/dispatcher process has the dependency provider loaded. The 2026-06-23 phase is authorized in `docs/live-autonomy-rollout-plan.md`; any broader Kanban-first live pilot still needs exact board/profile/workspace/artifact scope, provider-load/readiness evidence, dry-run/readback receipts before worker reliance, durable state/artifacts, and separate gateway-dispatch authorization. If credentials, scope, registry/repo permissions, Kanban board/profile prerequisites, gateway-dispatch authorization, or service exposure prerequisites are unavailable, report live validation as deferred rather than simulating success.
